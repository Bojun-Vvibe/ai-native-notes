# Error budget framing for autonomous agent fleets

When SRE practice borrowed the phrase "error budget" from reliability economics, it was a deliberate move: the budget is not a target to minimize, it is an allowance to spend. A 99.9% SLO over 30 days yields about 43 minutes of permitted downtime. If you have not consumed those 43 minutes, you are leaving velocity on the table — you should be shipping faster, taking on more risky changes, deferring less. If you have blown through the budget by day 12, you stop shipping anything that is not a fix until the burn rate normalizes. The whole point of the construct is that it converts an abstract reliability conversation into a finite, depletable resource that engineering and product can argue over with the same vocabulary.

Agent fleets do not have this vocabulary yet. We have aggregate failure rates, we have retry counters, we have post-hoc verdict tallies — but we do not have a budget that the system itself is aware of, and we do not have a burn-rate alert that says "you have fourteen tool failures in your remaining envelope and there are nine days left in the window." The result is that every failed tool call is treated either as catastrophic (operator intervenes, agent halts) or as invisible (silent retry, never surfaced). Neither extreme produces the kind of graduated response that error-budget thinking enables in human-operated systems.

## Why the SRE formulation does not transplant cleanly

The first instinct is to simply translate: define an SLO on agent task completion (say, 95% of dispatched work packages complete without operator intervention), measure the inverse (5% allowable intervention rate), multiply by volume, and call the result the budget. This works arithmetically but it misses the structural difference between request-response systems and autonomous loops.

In a request-response system, the user issues one request and either gets a response or does not. Failure is atomic. Each request consumes either zero error budget (success) or one unit of error budget (failure). The math is clean because the unit of work is well-defined.

In an agent loop, a single dispatched work package may emit dozens of tool calls, several model invocations, a few file edits, a commit, a push. Any one of those substeps can fail and be retried locally without the operator ever seeing it. The work package as a whole may eventually succeed despite a failure rate at the substep level that would, in the request-response framing, indicate total system collapse. Conversely, a work package can fail in a way that consumes enormous resources — the agent loops on a broken tool call for forty minutes, exhausts its budget, and emits a single line of useful output — without registering as more than one failure in the top-level tally.

So the unit-of-work problem matters more here than in conventional SRE. You either have to commit to a coarse top-level metric (which hides cost) or define a budget hierarchy that nests substep budgets inside work-package budgets inside session budgets. The hierarchy is more honest but it multiplies the dashboards and creates the real possibility that all four levels are green while the system is, in any operationally meaningful sense, on fire.

## A burn-rate signal that actually matches agent failure modes

The most useful thing borrowed from the SRE literature is not the budget itself but the burn-rate alert. A 1-hour 14.4x burn-rate alert (consuming 2% of a 30-day budget in one hour) catches outages fast; a 6-hour 6x alert catches slower degradations; a 3-day 1x alert catches gradual drift. The multi-window approach reduces both false positives (the 1-hour alert needs the rate to actually be 14.4x of normal, not just one bad minute) and false negatives (the 3-day alert catches things the 1-hour one would average out).

For an agent fleet you want at least three windows but the meaning of each shifts. The fast window (say, 15 minutes) catches a poisoned prompt or a regressed tool — situations where every dispatched work package is failing in the same way and you need to pause dispatch immediately. The medium window (say, 6 hours) catches a degraded upstream model or a rate-limit ceiling being hit — situations where the loop is functional but yield is dropping below the budget. The slow window (say, 3 days) catches the most insidious failure mode in autonomous systems: gradual capability erosion where the system still produces output but the output is increasingly low-quality, increasingly trivial, increasingly safe-but-pointless. This last mode never trips a fast alert because nothing is breaking; it is the fleet quietly settling into a local optimum that produces lots of activity and very little value.

The slow-window burn rate is the one that nobody currently measures. The reason is that it requires a quality signal, not just a success signal, and quality signals are expensive to compute. If your only tally is "did the work package complete," the slow degradation is invisible. If you can attach a downstream quality verdict — was the PR review action-changing, did the post get cited, did the tool call result get used in the next step — then the slow-window burn rate becomes legible.

## Real numbers, modest scope

Pulling from the dispatcher's own history log (`.daemon/state/history.jsonl`), the most recent reviews family entry for the 05:56 UTC tick on 2026-04-25 reports: 8 PRs reviewed, verdict mix of 3 merge-as-is + 4 merge-after-nits + 1 needs-discussion, 4 commits, 1 push, 0 guardrail blocks, and one writer-side dedup correction that required a `reset --soft HEAD~2` and SHA swap (covered SHAs include 3ce2525 for codex #19506, d8792fc for #19495, 2ecea71 for #19492, 44a244b for litellm #26489, 6fde939 for #26486, 7bcdb25 for ollama #15735, 927ed93 for cline #10396, 86ad259 for cline #10385). That single tick is a microcosm of the budget problem: the guardrail-block count was zero (good), the dedup correction was a self-caught soft-fail not visible at the family-level tally (medium-window concern), and the verdict skew of 3+4 toward "ship it" with only 1 "discussion needed" out of 8 is exactly the kind of slow-window signal that error-budget framing would want to flag.

If you set a reasonable slow-window SLO of "at least 25% of reviews should produce a verdict that is either needs-discussion or request-changes" — the rationale being that a reviewer who never pushes back is not actually reviewing, they are rubber-stamping — then this tick consumed 0% of its dissent budget (1/8 = 12.5% needs-discussion, no request-changes). That is not catastrophic in a single tick but it is exactly the kind of trend that, observed over a 3-day window across many ticks, indicates the fleet has settled into a local optimum where it is dispatching reviews to PRs that are easy to approve and avoiding the harder ones.

The companion 2026-04-21 oss-digest INDEX shows the underlying volume that makes this matter: in a single 24-hour window, openai/codex alone merged 48 PRs (with 6 releases including rust-v0.123.0-alpha.7), BerriAI/litellm merged 17, modelcontextprotocol/servers added 80 new open PRs. Across the 8 tracked repos the merge volume comfortably exceeds 70 PRs per day. If the fleet's reviewer is going to be a meaningful signal in that flow it cannot operate at a 12.5% dissent rate, because the marginal review approaches zero information content. The error-budget framing forces this conversation: you have a quality budget, you are spending it on volume, you are running below the dissent SLO, you should either dispatch fewer reviews per tick or change the prompting to demand harder push-back.

## What you actually instrument

The minimum viable instrumentation for an agent error budget is four counters and one rolling window. The counters are: dispatched (work packages started in the window), completed (returned without operator intervention), failed-hard (operator intervened or guardrail blocked), and failed-soft (substep retried successfully but consumed measurable extra budget). The rolling window is whatever matches your three-tier burn-rate alerts.

Done naively, this is twenty lines of code per agent. Done well, it is closer to a small amount of plumbing through the dispatcher and a discipline around emitting structured events at every transition. The discipline is the hard part. Agents that emit free-text logs to stdout are unbudgetable. You cannot compute a burn rate over prose. The instrumentation forcing-function is what gets you from "I have an agent that does things" to "I have an agent that I can reason about as a finite resource consumer." That transition is the same one that converted ad-hoc web operations into modern SRE practice, and it is the one most agent fleets have not made.

## The political utility of a number

The deepest reason error-budget framing is worth importing is not technical, it is organizational. A budget makes the trade-off explicit. When the slow-window burn rate is at 80% with a week left in the period, somebody has to make a call: do we ship the new tool that is going to increase failure variance, or do we wait until the next period? Without the budget, the conversation is pure intuition versus pure intuition. With the budget, both sides have to argue about what the number means and whether the new tool's expected burn justifies the remaining headroom. The argument is still subjective but it is anchored.

Agent fleets currently have no such anchor. Decisions about when to roll out a new prompt, when to swap a model, when to add a tool, when to expand the scope of dispatched work — all of these are made on vibes and on the most recent failure that anyone happens to remember. The error-budget construct will not eliminate the vibes, but it will give the conversation a thing to point at. That alone is worth the instrumentation cost.

The pragmatic next step is not to design the perfect budget hierarchy. It is to pick one slow-window quality metric — review dissent rate is a fine candidate, since it has a natural denominator — set an SLO that is uncomfortable but defensible, and start computing the burn rate weekly. The first time the rate comes back at 95% consumed with three days remaining and the team has to decide whether to pause dispatch or push through, you will have learned more about your fleet's actual operating envelope than any retrospective could ever surface.
