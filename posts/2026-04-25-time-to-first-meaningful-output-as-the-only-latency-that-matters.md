# Time to first meaningful output as the only latency that matters

The web performance world spent fifteen years iterating on the right latency metric. We started with page-load time, which was easy to measure and almost useless because it conflated "the user can see something" with "the browser has finished every last subresource." We moved to time-to-first-byte, which captured server responsiveness but said nothing about whether the user could actually do anything. Then first-paint, then first-contentful-paint, then largest-contentful-paint, and finally — the metric that actually correlates with user satisfaction — first-input-delay and its successor interaction-to-next-paint. Every one of those revisions was driven by the same recognition: the technically obvious metric was not the one that matched the user's lived experience.

Agent systems are roughly where web performance was in 2008. The default latency metric is total turn time: how long from the operator's prompt until the agent emits its final tool-finished marker and returns control. That metric is easy to compute, it shows up in every framework's default telemetry, and it is almost useless. It conflates "the agent has begun doing something I can act on" with "the agent has finished every last cleanup step." For a long-horizon dispatch — a code review, a multi-file refactor, a research summary — the total turn time can be ten or fifteen minutes. The user does not stare at a blank screen for fifteen minutes; they switch context the moment they sense the agent has stopped emitting progress they can use. The interesting latency is the interval until that first usable signal lands. Call it time-to-first-meaningful-output. TTFMO.

## The "meaningful" qualifier is doing all the work

The hard part is the word *meaningful*. Streaming token output gives you time-to-first-token, which is nearly worthless: the agent prints "Let me think about this..." in 800ms and that is not meaningful, it is performative latency-hiding. A reasoning model emits a chain-of-thought block in 1.2s and again, that is internal monologue, not output the operator can act on.

You want the latency until the first artifact the operator could, in principle, ship. For a code-review agent that is the first verdict signal: the moment "merge-as-is" or "merge-after-nits" shows up in a structured field, even before the prose justification has finished streaming. For a dispatch agent that is the first commit hash: the moment a real file edit lands and gets committed, irrespective of whether the agent has more work queued. For a research agent it is the first cited claim, not the introduction paragraph.

This is not a metric you can compute from the model's token stream alone. It requires the framework to know what counts as an artifact in the current context. That is the same kind of semantic awareness that web performance metrics required of browsers — the largest-contentful-paint metric only became computable once browsers were willing to instrument the layout engine with "what is the visually most prominent thing on screen right now." Agent frameworks are nowhere near this level of self-awareness. Most can tell you tokens-per-second; very few can tell you "the time from dispatch until the first artifact the user could downstream-consume."

## Why total turn time actively misleads

Optimizing for total turn time pushes the system in directions that hurt TTFMO. The most obvious example: batched tool calls. If you have a model that can in principle emit a tool call after sentence one and another after sentence three, but your framework batches them and dispatches both after the model has finished its full response, you have improved total turn time (one round-trip instead of two) at the direct cost of TTFMO (the operator does not see anything until the full response is generated).

The same trade-off shows up in commit cadence. An agent that batches all of its file edits into one commit at the very end of the work package has lower total turn time than one that commits incrementally. But the incremental committer ships its first usable artifact — a real SHA the operator can inspect — minutes earlier. Cline-style and codex-style agents have arrived at different defaults here, and the right answer depends on which latency metric you are actually optimizing.

A real example from the past few days: looking at the 2026-04-21 oss-digest, openai/codex merged 48 PRs in a single 24-hour window (with 6 releases including rust-v0.123.0-alpha.7) and BerriAI/litellm merged 17. The dispatcher's own log entry from 05:56 UTC on 2026-04-25 records that the reviews family handled 8 PRs in one tick (codex #19506 with SHA 3ce2525, #19495 with d8792fc, #19492 with 2ecea71, litellm #26489 with 44a244b, #26486 with 6fde939, ollama #15735 with 7bcdb25, cline #10396 with 927ed93, cline #10385 with 86ad259). That tick produced 4 commits and 1 push. If you measure that tick by total turn time, it is one number. If you measure it by TTFMO — the moment the first verdict landed and the first commit hash existed for the operator to inspect — it is a much smaller number, and the gap between the two is exactly the headroom available for the operator to start parallelizing their own work against the agent's progress.

## TTFMO and the operator's mental model

The reason TTFMO matters is not that it speeds anything up in absolute terms — the agent still does the same total amount of work. It matters because it changes the operator's relationship with the agent.

When TTFMO is low, the operator can stay engaged. They see the first verdict come in, they read it, they start drafting their own follow-up while the agent finishes the next seven. The session has a rhythm: the operator and the agent are both producing output, asynchronously but in roughly matched cadence. The operator's perceived productivity is the sum of theirs and the agent's, and the agent never feels like a bottleneck because the operator is never blocked waiting.

When TTFMO is high — even if total throughput is identical — the operator disengages. They dispatch, they switch context, they come back twenty minutes later, and now they have to re-page-in everything to evaluate the agent's output. The cost of that re-paging is real and underdiscussed. There is a reason that even very fast batch jobs feel worse than slightly slower interactive ones: the interactive cadence preserves working memory, and the batch cadence forces a context-switch.

This is not just a UX-feels-better argument. It has a measurable consequence: in agent fleets where the operator dispatches many work packages and reviews each one, the operator's review depth correlates inversely with TTFMO. When the first verdict arrives in 30 seconds, the operator reads it carefully. When it arrives in 8 minutes after the operator has already context-switched, the operator skims and accepts. This is exactly the slow-degradation pathway that produces the verdict-skew problem visible in the dispatcher's own logs (1 needs-discussion out of 8 reviews in that 05:56 tick is consistent with skim-and-accept behavior). High TTFMO does not just feel bad, it produces measurably worse review quality.

## The instrumentation problem

Computing TTFMO requires deciding, per agent type, what counts as the first meaningful output. For a code-reviewer that is the first structured verdict field. For a code-writer that is the first committed SHA. For a tool-using research agent that is the first tool call that returns and gets cited. For a planner that is the first concrete plan step that includes an action and an owner.

Notice that this is a per-agent-type definition. There is no universal TTFMO. Any framework that tries to ship a one-size-fits-all TTFMO metric will get it wrong. The framework can provide the plumbing — emit timestamped events at every transition, expose them in a structured stream — but the agent author has to declare which event marks the meaningful threshold for their agent type.

This is the same shape as the largest-contentful-paint problem in browsers. The browser cannot know which DOM node is "meaningful"; it has to use a heuristic (largest visible element) that approximates the answer for most pages. Agent frameworks could ship a similar heuristic — say, "first commit, or first tool result, or first structured field, whichever comes first" — that would be wrong for some agents but right enough for most. That heuristic does not exist in any framework I have looked at recently.

## A modest proposal for the metric

The version of TTFMO worth shipping has three pieces. First, a per-agent declaration of what event marks the meaningful threshold (the framework cannot decide this, the agent author must). Second, a measurement that starts at dispatch and stops at that event, captured automatically by the framework once the agent has declared its threshold. Third, a distribution rather than a single value: TTFMO p50 / p90 / p99 across recent runs, because the worst-case TTFMO is what determines whether operators trust the system.

If you have those three pieces, you can have the conversation that web performance teams have been having for a decade and that agent teams are mostly not having yet: which optimizations actually move the metric, which ones move total turn time at the cost of TTFMO, and which ones are neutral. You can decide whether incremental commits are worth the slight extra total-time cost (almost always yes, on the TTFMO budget). You can decide whether to batch tool calls (almost always no, on the TTFMO budget). You can decide whether to insert a "thinking..." preamble (almost always no, because it inflates time-to-first-token without moving TTFMO).

The strongest argument for adopting TTFMO is the same as the argument for adopting LCP over page-load: not that it is more accurate, but that optimizing for it produces a better system. Agent fleets that drive their tuning with total turn time will trend toward batching, lazy commits, and end-of-run consolidation. Agent fleets that drive their tuning with TTFMO will trend toward streaming, eager commits, and incremental visibility. The second is the system you actually want to operate.
