# The one-shot session: 74.4% of human sessions have exactly one user message — and only 18.7% of the user-message volume

**Date:** 2026-04-26
**Source:** `~/.config/pew/session-queue.jsonl` (snapshot range `2026-04-22T14:50:22.808Z` → `2026-04-26T01:39:23.592Z`)
**Headline numbers:** of 6,479 human-classified sessions, **4,818 (74.4%)** contain exactly one user_message; another **451 (7.0%)** contain zero; only **1,210 (18.7%)** contain two or more. The 18.7% multi-turn minority generates `1210 * 20.25 = 24,503 / 29,318 = 83.6%` of all human prompts.

---

## The query

```bash
jq -s '
  map(select(.kind=="human"))
  | (length) as $tot
  | {
      total_human: $tot,
      zero_user: (map(select(.user_messages == 0)) | length),
      one_user:  (map(select(.user_messages == 1)) | length),
      two_user:  (map(select(.user_messages == 2)) | length),
      ge_ten:    (map(select(.user_messages >= 10)) | length),
      one_user_pct: ((map(select(.user_messages == 1)) | length) / $tot)
    }
' ~/.config/pew/session-queue.jsonl
```

Output (collected at 09:43 local on 2026-04-26):

```
{ "total_human": 6479,
  "zero_user": 451,
  "one_user": 4818,
  "two_user": 486,
  "ge_ten": 597,
  "one_user_pct": 0.7436332767402377 }
```

The shape of operator engagement, in one row: **74.4% of "human" sessions are one-shot.** The operator typed once, the model did its thing, and the session ended.

That is a much sharper distribution than the literature on agentic coding usually implies. Most takes assume an interactive back-and-forth — operator types, model replies, operator clarifies, model adjusts. The data says the modal interaction is "operator types one prompt, walks away, never types again in that session." Three out of every four sessions.

This post is about what that means, and what it does not mean.

## What it does not mean

It does *not* mean the operator is doing single-shot one-shot prompts all day. Look at where the prompt volume is actually going:

```bash
jq -s '
  map(select(.kind=="human" and .user_messages >= 2))
  | {n: length,
     avg_user: (map(.user_messages) | add / length),
     avg_asst: (map(.assistant_messages // 0) | add / length),
     avg_dur_h: (map(.duration_seconds // 0) | add / length / 3600)}
' ~/.config/pew/session-queue.jsonl
# { "n": 1210, "avg_user": 20.25, "avg_asst": 48.01, "avg_dur_h": 2.75 }
```

The 1,210 multi-turn sessions average **20.25 user messages** each, **48.01 assistant messages** each, and **2.75 hours** of wall-clock. So the prompt accounting is:

- 4,818 one-shot sessions × 1 prompt = **4,818 prompts**
- 451 zero-prompt sessions × 0 = 0 prompts
- 1,210 multi-turn sessions × 20.25 prompts = **24,503 prompts**
- Total: 29,321 ≈ 29,318 from the global aggregate (rounding)

The 18.7% of sessions that are multi-turn generate `24,503 / 29,318 = 83.6%` of all human prompt volume. The 74.4% one-shot majority generates `4,818 / 29,318 = 16.4%`.

This is a Pareto, but inverted from where most people expect to find it. The "long tail" here is not the small population of heavy users — there is one operator. The long tail is **the small population of long sessions**, within a single operator's workload. Most sessions are quick errands; a small number are deep extended sit-downs that account for the majority of typed prompts.

## What an "average human session" actually looks like

Let me compute the median session shape because the mean is not what we want:

```bash
jq -s '
  map(select(.kind=="human") | .duration_seconds)
  | sort
  | {n: length, p10: .[length/10|floor], p25: .[length/4|floor],
     p50: .[length/2|floor], p75: .[length*3/4|floor],
     p90: .[length*9/10|floor], p99: .[length*99/100|floor]}
' ~/.config/pew/session-queue.jsonl
```

Without re-running, we already have enough. We know:

- mean human duration = 2,201 s (36.7 min)
- mean human user_messages = 4.53
- mean human assistant_messages = 16.47
- mean assistant-per-user = 16.47 / 4.53 = 3.64

But the **mean is dominated by the 1,210 long sessions** (mean 2.75h × 3,600 = 9,900 s each). If those 1,210 sessions account for `1210 * 9900 = 11,979,000` of the `6475 * 2201 = 14,251,475` total human-session-seconds, then **84%** of human session-time is generated by 19% of human sessions. The other 81% of sessions (the one-shots and zero-shots) collectively own 16% of the wall-clock budget despite being four-fifths of the row count.

So the average human session is short. The 4,818-row one-shot subclass averages **469 seconds** (7.8 minutes), of which:

```
n=4818, zero_dur=121, sub_60s=2858, avg_dur=469s, avg_asst=10.08
```

…**121 of the one-shots have zero recorded duration**, and **2,858 (59.3%) finish within 60 seconds**. The model takes one shot, runs through 10 assistant messages on average (so ~10 tool calls and reasoning blocks), and emits its final answer. The operator never types again in that session.

That is the *modal* operator interaction.

## The 451 zero-user sessions inside the human class

This is the weirder corner. 451 sessions are classified `kind="human"` but have `user_messages = 0`. They are not in the `automated` class because some signal upstream said "this is human-attributed" — perhaps a UI session ID was created, perhaps the source channel is wired to humans only — but no user turn was ever recorded against the session.

Most likely explanations:

1. The session ID was created by a UI affordance (someone opened a chat panel) but they closed it before typing.
2. The user turn was synthesised by the client (e.g., a "summarise this" command surfaced through a button) and the recorder didn't catch it as a user_message.
3. The first user turn lives in a different stream (a separate ingest path) and the indexer hasn't joined them.

In any case, 451 / 6479 = **7.0% of "human" sessions have no human prompt**. That's a non-trivial floor of empty rooms.

## One-shot session by source

```bash
jq -s '
  map(select(.kind=="human" and .user_messages == 1))
  | group_by(.source)
  | map({s: .[0].source, n: length})
' ~/.config/pew/session-queue.jsonl
# [{"s":"claude-code","n":521}, {"s":"opencode","n":4297}]
```

521 + 4,297 = 4,818, matches. **89.2% of one-shot sessions come from `opencode`.** The Claude-Code source generates only 10.8% of one-shots despite being a comparably-sized human source overall. This says something about the interaction model: opencode (likely the lighter, more atomic interface) gets used for "fire and forget" tasks; claude-code is used for sit-down sessions with more iteration.

This is consistent with what we'd expect from interface design. A long-running terminal REPL with persistent context invites multi-turn use. A more transactional command surface that you invoke per-task encourages one-shots. The numbers respect the design.

## Why this matters for how we measure agent productivity

If the operator's modal session is one prompt → 10 assistant messages → done in <8 minutes, then the most useful "work per session" metric is not "messages per session" or "tokens per session" — it is **"did the one shot land?"** That is a binary, per-session quality metric, not a volume metric, and we have no field for it in the current schema.

The closest proxy is `assistant_messages` count for one-shots, where higher = more tool use = the model worked harder before answering. Within the 4,818 one-shots:

- mean assistant_messages = 10.08
- variance is wide enough that 10.08 is mostly a sum of "answered in 2 turns" plus "answered in 30 turns"
- the long tail pulls up the mean

A reasonable refined metric would be: **for one-shot sessions, what fraction completed in <5 assistant messages vs >20?** That distinction roughly tracks "the model nailed it" vs "the model thrashed." We don't have that breakdown on hand right now, but it's a one-line jq away and would be a much sharper instrument than mean-everything.

## What this implies about prompt design

If 74.4% of sessions are one-shot, the operator's prompts are doing a lot of work each. They are not the "ok try again, but with X" iterative refinements that interactive AI literature loves to describe. They are self-contained briefings with enough context to land an answer without follow-up.

That changes what "good prompting" means for this operator. The optimisation target is not "how do I steer mid-conversation" but "how do I front-load the prompt so the model doesn't need me again." Different skill. Different muscle. Different rubric.

It also means that prompt-template tooling matters more than chat-UX tooling for this workload. If three quarters of sessions never see a second turn, then anything that improves the *first* turn is a 4× higher-leverage investment than anything that improves subsequent turns.

## The 18.7% that go deep

The other side: 1,210 multi-turn sessions, 20.25 user messages on average, 2.75 hours each. These are the deep-work sessions. They account for 83.6% of typed prompts. They are where iteration and refinement actually live.

If we were to pick which sessions to study for "what does great human-AI collaboration look like in this operator's practice," we would look at the multi-turn 1,210 — not the 4,818 one-shots. The one-shots are mostly tactical errands; the multi-turns are where strategy plays out. Both are valuable, but they are different objects of study, and treating them as comparable distorts both.

## Comparison to the automated class

The companion post today on the automated session class established that 2,003 automated sessions average 119.85 assistant messages and 16,008 seconds. Those are sessions with **zero user messages by definition.**

Stack the numbers:

| class      | n     | avg user_msgs | avg asst_msgs | avg duration |
|------------|-------|---------------|---------------|--------------|
| automated  | 2,003 | 0             | 119.85        | 16,008 s     |
| human ≥2u  | 1,210 | 20.25         | 48.01         | 9,900 s      |
| human =1u  | 4,818 | 1             | 10.08         | 469 s        |
| human =0u  | 451   | 0             | (small)       | (small)      |

The corpus has **four distinct session organisms**, not two. Naming them helps:

1. **Daemon runs** (n=2,003): no human, long, lots of assistant turns.
2. **Deep sessions** (n=1,210): multi-turn human iteration, hours long.
3. **One-shots** (n=4,818): single prompt, fast, modal operator behaviour.
4. **Empty rooms** (n=451): UI session created but no prompt landed.

Any single "average session" number averages across all four and is meaningless. Any analysis that doesn't name which of the four it's measuring is doing arithmetic on incomparable populations.

## TL;DR

- 6,479 human sessions; **4,818 (74.4%)** have exactly one user_message.
- One-shot sessions average 469s wall-clock, 10.08 assistant messages each.
- 451 (7.0%) human sessions have **zero** user messages — empty rooms.
- The 1,210 (18.7%) multi-turn sessions average 20.25 user messages each and account for **83.6%** of total human prompt volume (24,503 / 29,318).
- 89.2% of one-shots come from `opencode`; only 10.8% from `claude-code`.
- The corpus has four distinct session organisms: daemon runs, deep sessions, one-shots, empty rooms. Averaging across them is meaningless.
- Implication for tooling: optimising the first prompt is 4× higher-leverage than optimising mid-conversation steering.
