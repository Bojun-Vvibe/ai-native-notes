# Conversation compaction strategies: summary vs sliding window vs hierarchical

**Date:** 2026-04-25
**Tags:** context-window, compaction, agent-runtime, memory, cost

Every agent that runs longer than five turns eventually has to
delete some of its own history. The context window is finite, the
prompt-cache prefix is sacred, and the user keeps wanting to
reference "what we decided two hours ago." The mechanism that
lives at this intersection is *conversation compaction*, and the
choice of strategy is one of the most consequential design
decisions in an agent runtime — second only, maybe, to tool-call
parsing.

There are three families in the wild: **summary-based**,
**sliding window**, and **hierarchical**. They are not
interchangeable. Each preserves a different invariant, fails in a
different shape, and prices itself differently against the
provider's billing model.

## Why compaction even exists

The naive answer is "context window limits," which is true but
incomplete. A 200K-token window is plenty for most sessions; the
real reasons the runtimes compact aggressively are:

1. **Cost per turn grows linearly with prompt length** even
   under prefix-cache pricing, because the *uncached suffix* (the
   user's new message + the model's last response) gets longer
   too — every appended assistant turn is a cache miss for the
   *next* request.
2. **Latency grows with prompt length** in a way that is harder
   to amortize than cost. Time-to-first-token under prefix-cache
   on a long uncached suffix can be measured in seconds. Operators
   notice latency before they notice cost.
3. **Models degrade on long contexts.** This is the boring,
   well-known one. Past ~80K tokens of conversation history,
   instruction-following gets noticeably mushier on every model
   family currently shipping. Compaction is partially a quality
   intervention, not just a cost intervention.

So the runtime needs a strategy. The three families:

## Strategy 1: summary-based ("compact when full")

Run the conversation forward until you hit a high-water mark
(typically 60–80% of the context window), then ask the model to
produce a structured summary of the conversation so far, then
**replace** the older turns with that summary and continue. The
canonical implementation looks roughly like:

```
[ system prompt ]
[ <COMPACTED SUMMARY> turns 1-37 ]
[ assistant turn 38 ]
[ user turn 39 ]
[ assistant turn 40 ]
...
```

This is what most user-facing agents default to. It is
operationally pleasing — the conversation stays coherent, the
user can still ask "what did we decide about X" and get a sensible
answer, and the summary is itself a model artifact so the runtime
doesn't have to write a heuristic.

The pitfalls are well-rehearsed but worth naming:

- **The summary destroys the prefix cache.** Every compaction
  rewrites the entire prompt prefix, which means the next request
  is a 100% cache miss. If your provider charges 10× more for
  uncached input than cached input (typical), the turn
  *immediately after* a compaction is the most expensive turn in
  the session. Some runtimes hide this by compacting eagerly
  during a quiet moment (model is generating) so the user
  doesn't *feel* the latency, but the bill still arrives.
- **The summary loses precise quotes.** "We decided to use
  PostgreSQL because the rate limit on the analytics service was
  600 req/min" gets summarized to "decided to use PostgreSQL for
  the analytics layer" — which is true but loses the number, the
  reason, and the constraint that justifies the decision. Two
  turns later the agent helpfully suggests Redis.
- **The summary is generated by the same model that's about to
  forget the original.** This is a recursive trust problem. The
  model writing the summary cannot tell you what it does *not*
  know is important. So the summary is biased toward the
  topics the model thought were worth talking about, which are
  exactly the topics it least needed reminded of.

A common refinement: have the runtime, not the model, dictate
*which* turns to summarize and which to keep verbatim. Always
keep the system prompt, the most recent N turns, and any turns
containing tool calls whose results are referenced later. Send
only the in-between to the summarizer. This addresses two of the
three pitfalls but not the cache-bust.

## Strategy 2: sliding window ("forget the oldest")

Keep the system prompt + the most recent N turns; drop everything
else with no replacement. The transcript looks like:

```
[ system prompt ]
[ ...nothing... ]
[ assistant turn 30 ]
[ user turn 31 ]
[ assistant turn 32 ]
...
```

The window slides forward one turn at a time as new turns arrive.
This is what many *programmatic* agents (the ones with tight
loops, no human in the chair) use, and it is also what the more
honest interactive agents fall back to when summarization is too
expensive or too risky.

Sliding window's virtue is that **the prefix cache stays warm**.
The system prompt at the front is unchanged across turns, so
every request after the first is a cache hit on at least the
system-prompt span. Even the older turns inside the window can
remain cached if the runtime is careful about how it stitches
them. This makes sliding window dramatically cheaper per turn
than summary-based for long sessions.

Its sin is the obvious one: the agent forgets, hard. There is no
graceful "I vaguely recall we discussed PostgreSQL" — there is
only "Postgre-what?" The agent will cheerfully re-derive a
decision the user thought was settled, and the user will be
furious.

A real data point worth pinning here: `anomalyco/opencode`
v1.14.19 (released 2026-04-20T07:21:48Z) renamed its
sliding-window knob from whatever it was called before to
**`preserve_recent_tokens`**, which is the most honest variable
name in the entire ecosystem. It says exactly what the strategy
does: preserve a fixed number of recent tokens, and discard the
rest. The same release improved the session compaction logic in
PR #23870 — meaningfully, the change is from a turn-counted
window to a token-counted window, because turns vary wildly in
length and a 30-turn window can be either 4 KB or 60 KB depending
on how chatty the tool calls were.

The 2026-W17 weekly digest covering 2026-04-20 → 2026-04-26
notes that opencode shipped three releases in this window
(v1.14.19, v1.14.20, v1.14.21) and that the breaking-change risk
"concentrates around the v1.14.21→v1.14.22 boundary
specifically" — most of the schema migration that happened
between those two tags touched the same compaction surface, which
is its own argument for keeping the strategy as simple as
sliding window.

## Strategy 3: hierarchical ("summary of summaries")

The third family tries to have it both ways. Keep a sliding
window of recent turns *verbatim*. As turns fall out of the
window, push them into a "level-1 summary." When the level-1
summary itself gets long, summarize it into a "level-2 summary."
And so on. The transcript looks like:

```
[ system prompt ]
[ <L2 SUMMARY> turns 1-200 ]
[ <L1 SUMMARY> turns 201-280 ]
[ assistant turn 281 ]
[ user turn 282 ]
...
```

This is what the more ambitious agent runtimes are converging on,
and it is also where the most subtle bugs live.

The appeal is real: distant history is preserved at low fidelity,
recent history is preserved at high fidelity, and the cost of
each new turn does not grow without bound. The cache-bust still
happens on summary creation but only for the affected layer; the
upper layers stay cached for many turns at a time.

The trouble is the **summarization-of-a-summary degrades
non-linearly**. By the third level, the L3 summary is paraphrasing
a paraphrase of a paraphrase, and the model has no way to recover
the original signal. Specific facts (numbers, identifiers, code
snippets) are the first casualties. If you are building an agent
where someone might ask "what was the SHA we agreed to revert to
yesterday," hierarchical summarization will lie to them.

A common mitigation: *anchor* specific value classes (filenames,
URLs, version strings, commit SHAs, tool-call identifiers) in a
side table that survives all levels of summarization, and
re-inject them into the active context on demand. Some runtimes
implement this as a structured "facts" sidecar; others as a
simple regex extractor that hoists anything matching `[a-f0-9]{7,40}`
into a permanent block. Crude, but effective.

## What the W17 release notes tell us about the state of the art

The 2026-W17 weekly digest (window 2026-04-20T00:00:00Z → 2026-04-
26T23:59:59Z, covering 8 repos) records `charmbracelet/crush`
v0.62.0 as the headline release, with a **98% reduction in tool-
description tokens by default** (PR #2679). That number is about
*tool-definition* compaction, not conversation compaction, but it
is the same family of intervention: the longest, most-amortizable
part of the prompt got compressed, freeing budget for the
volatile suffix. The same release added a `fetch`/`view`
truncation cap at 100 KB (PR #2671) — a hard ceiling that
prevents any single tool result from blowing the entire window
budget, which is the upstream cause of forced compactions.

`openai/codex` 0.123.0 stable (released 2026-04-23T01:26:06Z)
shipped 186 PRs over the W17 window and promoted both `codex_hooks`
(PR #19012) and the rebranded "auto-review" reviewer (PR #19063)
to stable. Hooks gained MCP-tool support in PR #18385 and config-
file declaration in PR #18893. Why this matters for compaction:
hooks let the runtime intercept the conversation at well-defined
points, which is exactly where a smart compaction strategy needs
to insert itself. Codex's compaction is moving from
"automatically when full" toward "at hook-defined boundaries the
operator controls," which is a real architectural shift.

`BerriAI/litellm` cut v1.83.11-nightly with adaptive routing in
PR #26049 and per-team-member spend tracking in PR #26195. The
adaptive routing piece is relevant here because *which model you
route to* changes the compaction calculus: a model with a 200K
window and aggressive prefix-cache discounts can sliding-window
cheaply, while a smaller fallback model needs summary-based
compaction to fit at all. A routing layer that decides per-turn
which model handles this turn cannot use a single fixed
compaction strategy across the session.

## Choosing in practice

A pragmatic rubric, distilled from watching the OSS agents
diverge over the last two months:

- **Programmatic agents with tight loops, no human:** sliding
  window with a token-counted budget. `preserve_recent_tokens`-
  style. Cache-friendly, deterministic, easy to reason about, and
  the agent's "forgetfulness" is acceptable because the loop
  itself is short.
- **Interactive coding agents with humans in the chair:**
  hierarchical with anchored facts. The user *will* ask about
  something from an hour ago. Sliding window will rage-quit them.
  Pure summary-based will make up plausible nonsense. Hierarchical
  with a SHAs-and-filenames sidecar is the least-bad option.
- **Long-running planning/research agents:** summary-based with
  *operator-supervised* compaction. Compact at moments the
  operator chooses (end of a phase, between research questions),
  not at moments dictated by token count. Pay the cache-bust
  cost on a schedule the operator can predict, not at the worst
  possible moment in the middle of a turn.
- **Multi-tenant proxy / routing layers (LiteLLM-shaped):** do
  not compact at this layer at all. Pass the full transcript and
  let the consumer's runtime decide. A proxy that silently
  compacts is a proxy that silently changes the meaning of the
  conversation, and that bug is brutal to diagnose downstream.

## The unsexy operational notes

Three things every compaction implementation eventually has to
add, regardless of strategy:

1. **A "compaction event" log entry**, structured, with the
   pre- and post-token counts and the strategy that fired. When
   the user complains "the agent forgot what we said," the only
   useful debugging artifact is a timeline showing exactly when
   compaction ran and what got dropped. Without this you are
   guessing.
2. **A way to *un*-compact** for a single turn — i.e. include
   the original (uncompacted) span just for the next request, at
   the cost of one cache miss, when the user explicitly asks
   about old history. This is rare but it is the difference
   between "the agent forgot" (bad) and "the agent had to look
   it up" (fine).
3. **A test fixture** that runs a 100-turn synthetic conversation
   and asserts that specific facts (a number, a filename, a
   commit hash) injected at turn 5 survive into the prompt at
   turn 95. Hierarchical strategies in particular fail this test
   silently and the regression appears weeks later.

Compaction is one of those subsystems where every design choice
is a tradeoff between *cost*, *latency*, *coherence*, and *user
trust*, and there is no Pareto-optimal answer. Pick a strategy,
write down what it preserves and what it discards, and put both
in the operator's documentation. The worst compaction strategy
is the one nobody can describe in one sentence.
