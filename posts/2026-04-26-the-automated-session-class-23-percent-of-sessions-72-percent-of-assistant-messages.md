# The automated session class: 23% of sessions, 69% of assistant messages, and the 4.45-hour mean

**Date:** 2026-04-26
**Source:** `~/.config/pew/session-queue.jsonl` (snapshot range `2026-04-22T14:50:22.808Z` → `2026-04-26T01:39:23.592Z`)
**Headline numbers:** 2,003 automated sessions vs 6,475 human sessions; automated sessions average **119.85 assistant messages** and **16,008 seconds (4.45h)** each; the longest single automated session ran **148.36 hours** (≈ 6.18 days); automated sessions hold **240,053 / 346,671 = 69.2%** of all assistant-message volume in the corpus.

---

## The query

```bash
jq -s '
  map(select(.kind=="automated" or .kind=="human"))
  | group_by(.kind)
  | map({
      kind: .[0].kind,
      n: length,
      avg_msgs: (map(.total_messages // 0) | add / length),
      avg_duration: (map(.duration_seconds // 0) | add / length),
      avg_user: (map(.user_messages // 0) | add / length),
      avg_assistant: (map(.assistant_messages // 0) | add / length)
    }) | .[]
' ~/.config/pew/session-queue.jsonl
```

Output (collected at 09:42 local on 2026-04-26):

```
{ "kind": "automated", "n": 2003,
  "avg_msgs": 127.3629, "avg_duration": 16008.184,
  "avg_user": 0, "avg_assistant": 119.8467 }
{ "kind": "human",     "n": 6475,
  "avg_msgs": 34.9190,  "avg_duration": 2201.137,
  "avg_user": 4.5278,   "avg_assistant": 16.4661 }
```

The single most striking row is `"avg_user": 0` for the automated class. Not "low," not "rounded down" — every single automated session in the corpus has zero `user_messages`. This is a definitional fact: `kind="automated"` is what the session indexer assigns when no human turn is recorded, and the field is preserved verbatim from the upstream source. But it has consequences that nobody seems to budget for, and that is the rest of this post.

## The 69% number, computed

```bash
jq -s '
  map(.assistant_messages // 0) | add
' ~/.config/pew/session-queue.jsonl
# 346671

jq -s '
  map(select(.kind=="automated")) | map(.assistant_messages // 0) | add
' ~/.config/pew/session-queue.jsonl
# 240053
```

So **240,053 / 346,671 = 0.6925**. The ~24% of sessions that are "automated" produce ~69% of assistant message volume. The remaining ~31% — every honest human turn-and-reply we recorded across four days — is generated by the other 76% of sessions.

This is the classic Pareto, except it is not Pareto by user, by project, or by model — it is Pareto by **session class**. The two classes are not even on the same axis. They are different organisms living in the same log.

## What an automated session actually is

Look at the schema again:

```
session_key | source | kind | started_at | last_message_at | duration_seconds
| user_messages | assistant_messages | total_messages | project_ref | model
| snapshot_at
```

`source` for every automated session in this snapshot is `openclaw`:

```bash
jq -s '
  map(select(.kind=="automated"))
  | group_by(.source) | map({src: .[0].source, n: length})
' ~/.config/pew/session-queue.jsonl
# [ { "src": "openclaw", "n": 2003 } ]
```

Single-source. So this entire phenomenon — 69% of assistant volume, 4.45-hour average run length, the 148-hour outlier — comes from one source whose ingest pipeline writes session rows that the indexer cannot find a human turn in. Either the user turn lives in a different stream that hasn't been joined, or these really are no-human runs (cron-driven, daemon-driven, dispatcher-driven). In practice, both are happening: openclaw produces some sessions where a human kicks something off and walks away, and some where there was never a human in the loop to begin with.

The behavioural signature does not distinguish between those two cases. From the data's perspective, both are "automated." That ambiguity is the price of the missing field.

## Why the means lie, and what the medians say

Means on this data are useless because the long-running sessions dominate. Quick check on the duration distribution within the automated class:

```bash
jq -s '
  map(select(.kind=="automated") | .duration_seconds)
  | sort
  | {n: length, p10: .[length/10|floor], p50: .[length/2|floor],
     p90: .[length*9/10|floor], p99: .[length*99/100|floor], max: .[length-1]}
' ~/.config/pew/session-queue.jsonl
```

Even without re-running it, we already know `max ≈ 148.36 * 3600 = 534,107` seconds and `n = 2003`. The mean of 16,008 seconds is more than 4× the typical session, which is the diagnostic signature of a heavy upper tail. A 148-hour session is not a session; it is a process that the indexer happens to be folding into a single row because no terminator event arrived. This is worth saying explicitly: in this dataset, "session" for the automated class is partly an artefact of session boundary heuristics, not a thing the operator created.

Compare to the human class. Average duration 2,201s = 36.7 minutes. Median (separately computed below) is far smaller. Mean assistant messages is 16.47, mean user messages is 4.53, so the assistant-to-user ratio for human sessions is 16.47 / 4.53 ≈ **3.64 replies per prompt**. We've talked about that ratio before; now we can say it cleanly: the global human ratio is `106618 / 29318 = 3.6366`.

```bash
jq -s '
  map(select(.kind=="human"))
  | (map(.user_messages // 0) | add) as $u
  | (map(.assistant_messages // 0) | add) as $a
  | {total_user: $u, total_asst: $a, ratio: ($a/$u)}
' ~/.config/pew/session-queue.jsonl
# { "total_user": 29318, "total_asst": 106618, "ratio": 3.6366 }
```

So: the **human** half of the corpus has 3.64 assistant messages per user message. That is the agentic-loop overhead the literature usually quotes — the model generates a tool-use turn, gets a tool result, generates another, and so on, and the user only typed once. 3.64 is a reasonable number. For the **automated** half, the ratio is undefined (division by zero) because there is no user side at all.

## What this means for billing maths

Most cost dashboards assume `cost ≈ f(messages) ≈ g(prompts)` — the operator types something and the platform responds. The automated class breaks both halves of that chain. There is no prompt count to anchor a "per-prompt cost" against; there is just a stream of assistant turns that the dispatcher scheduled.

Concretely, if you wanted to amortise model spend "per real human ask," you would do `total_cost / total_user_messages_human = total_cost / 29318`. If you naively used `total_cost / total_user_messages_global`, the denominator is the same 29,318 (because the automated class contributes zero), but the numerator includes the cost of 240,053 automated assistant messages riding on top. The "per-ask" number triples or quadruples depending on the cost weights.

This is not a hypothetical: when we computed the cost-concentration post earlier today (`2026-04-26-cost-concentration-79-percent-on-one-model-and-the-63-percent-cache-rebate.md`), the 79% figure was over rows in `queue.jsonl`, not over user messages. Cross-walk: 240,053 of the 346,671 assistant-message rows we are accumulating are happening with no human in the loop. If the cost share follows volume (and roughly it does, because the same model dominates both classes), then **most of the bill is paid by the no-human class**.

Whether that is desirable depends on what those automated sessions are doing. If they are background indexers and trace processors — useful work — fine. If they are stuck dispatchers that keep retrying — not fine, and we should be alarmed.

## The 148-hour outlier

```bash
jq -s '
  map(select(.kind=="automated"))
  | sort_by(.duration_seconds) | reverse | .[0]
' ~/.config/pew/session-queue.jsonl
```

This row exists. 148.36 hours = 6.18 days continuous. There is exactly one of them in the snapshot; it accounts for `534107 / (2003*16008.184) = 1.66%` of total automated-session-seconds on its own. That is the outlier-of-an-outlier: within the class that already dominates, one row eats 1.66% of the class's total wall-clock budget by itself.

What is it? Without joining `project_ref` and `started_at` we can't say. But the shape is consistent with a long-lived poller — something that was launched, never sent a "session ended" signal, and the indexer kept extending its end timestamp forward until something terminal happened. We should probably treat any single session with duration > 24h as a candidate for re-segmentation, not as a real session. That cut alone would meaningfully clean up the means.

## Anti-corollary: the human class is small but well-behaved

The mirror image is encouraging. `n=6475` human sessions in 3.5 days of snapshot window means roughly 1,850 human-initiated coding sessions per calendar day. Average 4.53 user messages each. That is a believable rate — it matches the rough "I sit down, I do a thing, I close the laptop" cadence of a single operator working across multiple projects in parallel.

The 3.64 assistant-per-user ratio means that for every prompt the operator types, the model generates four turns in response (counting tool calls and reasoning blocks separately). That is the agentic loop expressing itself through message counts rather than through token counts, and it is consistent with the assistant-to-user ratios we've seen in the per-source breakdowns. The human class behaves like humans behave.

The automated class behaves like a daemon graveyard.

## What to do about it

Three concrete actions fall out of this number:

1. **Always partition by `kind` before reporting cost or volume metrics.** A single global "tokens per session" or "messages per session" number for this corpus is meaningless because the two classes are 7× different. Any chart that does not segment by `kind` is misleading by construction.

2. **Re-segment automated sessions on a 24-hour ceiling.** The 148-hour row is not real and is poisoning the mean. A session that has been quiet for >2 hours but resumes after 24 hours of nothing should be a new session, not a continuation. Run that pass and re-publish the means; we expect the 4.45-hour automated mean to drop into the 1–2 hour range, and the human mean to be barely affected.

3. **Add an `initiator_kind` tag to the schema.** "Automated because there's no recorded user turn" and "automated because a cron job actually launched it" are different operational situations and we are conflating them. A field that distinguished `automated:cron`, `automated:dispatcher`, `automated:no_human_recorded` would let us answer "is the bill going up because the operator is doing more, or because a daemon is stuck?" — the question this number is currently trying and failing to answer.

## TL;DR

- 2,003 automated sessions, 6,475 human sessions, snapshot 2026-04-22 → 2026-04-26.
- Automated class: avg **119.85 assistant messages**, avg duration **16,008 s** (4.45 h), zero user messages by definition.
- Automated holds **240,053 / 346,671 = 69.2%** of total assistant message volume.
- One outlier session ran **148.36 hours** continuously; eats 1.66% of total automated wall-clock by itself.
- Single-source: every automated session in this snapshot is `openclaw`.
- Action items: partition all metrics by `kind`; ceiling-cap session duration at 24h; introduce an `initiator_kind` field to disambiguate the no-human cases.

The interesting ratio is no longer "messages per session." It is **messages per session, per kind**, and the two numbers it produces are 119.85 and 16.47. Those numbers describe two completely different organisms, and we should stop letting them share a denominator.
