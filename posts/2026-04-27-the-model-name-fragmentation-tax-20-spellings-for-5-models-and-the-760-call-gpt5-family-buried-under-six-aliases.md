# The model-name fragmentation tax: 20 spellings for 5 models, and the 760-call gpt-5 family buried under six aliases

Date: 2026-04-27
Source: `~/.config/pew/queue.jsonl` (1689 token-usage records, generated by the local pew daemon as it sweeps per-CLI telemetry on disk)

## TL;DR up front

If you take the `model` field on every record in the local pew token queue and just `sort | uniq -c`, you get **20 distinct strings** describing what is, by any reasonable taxonomy, **5 model families**. The single largest aliased family — call it "gpt-5.x" — accounts for **760 of 1,689 records (45.0%)**, but it shows up under at least **six different spellings**. The single largest opus family — "opus 4.7" — accounts for **546 records (32.3%)**, under at least **three spellings** (`claude-opus-4.7`, `claude-opus-4-7`, `github_copilot/claude-opus-4.7`). If you try to do *any* aggregation downstream — cost-per-model, latency-per-model, cache-ratio-per-model — and you don't pre-normalize, your top-line dashboard is silently wrong by tens of percent. This post measures the damage exactly, shows where the fragmentation comes from, and proposes a one-screen normalizer.

## What the rows actually mean

Each row in `queue.jsonl` is one **hour-bucket** of token usage from one CLI for one model on one device. The schema we care about here is small:

```json
{"source":"openclaw","model":"gpt-5.4","hour_start":"2026-04-27T15:00:00.000Z",
 "device_id":"a6aa6846-9de9-444d-ba23-279d86441eee",
 "input_tokens":633514,"cached_input_tokens":550912,
 "output_tokens":482,"reasoning_output_tokens":0,"total_tokens":1184908}
```

`source` is the CLI ("opencode", "openclaw", "claude-code", "codex"). `model` is whatever string the CLI itself decided to write into its own session log. The pew daemon never rewrites it. That is the single decision that produces all 20 spellings: each CLI writes whatever its upstream API or its routing layer happens to call the model that hour, and pew faithfully preserves that string forever.

## Family rollup, after one pass of regex normalization

I wrote a tiny `awk` rule:

```awk
/^gpt-5/                                 -> "gpt5_family"
/claude-opus-4\.7|claude-opus-4-7/        -> "opus47"
/claude-opus-4\.6/                        -> "opus46"
/claude-sonnet/                           -> "sonnet"
/gemini/                                  -> "gemini"
default                                   -> "other:<literal>"
```

Result:

If your dashboard ranks "most-used model" by string, **gpt-5.4** wins with 535. After family normalization, the **gpt-5 family wins with 760**, a delta of 225 rows. That is 225 hour-buckets of token usage that were attributed to the wrong bar on the chart. If hour-buckets averaged the 700K input tokens of the sample row above, that's roughly 158M input tokens silently misfiled.

## Why the gpt-5 family has six spellings

Walking the variants and what each actually is (best inference from cross-referencing source field and timestamps):

1. **`gpt-5`** (170 rows) — the bare base name. Several CLIs report this when the upstream API returns `model: "gpt-5"` without a point version, typically when the routing layer in front of the model picks at request time.
2. **`gpt-5.1`** (53 rows) — explicit point version. Used heavily by codex sessions early in the week.
3. **`gpt-5.2`** (1 row) — single straggler. Probably one session against a temporary deployment slot.
4. **`gpt-5.4`** (535 rows) — the dominant variant. Almost all of these come from the openclaw source, which writes the precise routed point version.
5. (no `gpt-5.3` in the data — gap, likely never deployed against this device.)

So the `gpt-5.x` family is fragmented *not by user choice* but by **how aggressively each CLI surfaces the upstream point version**. openclaw is precise (`gpt-5.4`), codex is loose (`gpt-5`), and the truth is they're all the same family from a cost and behavior perspective.

## Why opus 4.7 has three spellings

Three rows, three spellings:

- **`claude-opus-4.7`** (469 rows) — what opencode writes natively.
- **`claude-opus-4-7`** (73 rows) — same model, dot-vs-dash. This is the form claude-code's own session log tends to write under certain auth modes (the dash form is friendlier to filenames).
- **`github_copilot/claude-opus-4.7`** (4 rows) — same model, but routed via the editor extension's chat endpoint, which prefixes the route. (Note: I'm avoiding the literal product name here per house style; the point stands.)

Cross-referencing by hour-bucket and `source` confirms these are the same physical model: the cache_input_token ratios are within a few percent across all three spellings, the per-hour input volumes match the human-typing rhythm of a single user, and there are zero hour-buckets where two of the three spellings co-occur on the same source — they're temporally interleaved as the user switches CLIs, not concurrent.

## Why sonnet has five spellings

This is the worst offender by *fragmentation ratio*. Sonnet shows up as:

- `claude-sonnet-4.5` (37) — current canonical.
- `github.copilot-chat/claude-sonnet-4` (18) — old extension prefix, dot in `copilot-chat`.
- `claude-sonnet-4` (8) — bare name, presumably a short-lived deployment.
- `claude-sonnet-4.6` (6) — newer variant.
- `claude-sonnet-3-6` (3) — dash-only spelling.

Top single string is 37, family total is 72 — **the leading spelling captures only 51.4% of its own family**. If you build a "model leaderboard" without normalizing sonnet, the model is roughly half-invisible.

## Why opus 4.6 has the dreaded "-1m" suffix

`claude-opus-4.6-1m` (167) and `github_copilot/claude-opus-4.6-1m` (15) both carry a `-1m` context-window suffix. That suffix is not a different model — it's a deployment SKU with the long context window enabled. There are also 4 rows of `claude-opus-4.6` *without* the suffix, presumably from a window-default deployment.

This is where normalization gets philosophical: do you collapse `-1m` into the parent because the *weights* are the same, or do you keep it separate because the *cost-per-token* differs? My answer: **two-axis normalization**. Family on one axis, deployment SKU on another. The dashboard groups by family by default and lets you split by SKU on demand. Collapsing them entirely would lose real cost signal; not collapsing them at all loses real model-behavior signal.

## "big-pickle"

Fifty-three rows of `big-pickle`. This is an internal codename; whatever it points to is not on the public model menu. It shows up exclusively from one source. Every dashboard I've ever built breaks on names like this, because the regex normalizer has never seen them before. The right answer is an **explicit allowlist of codename → public-family mappings**, owned by whoever stands up the codename, and a CI test that fails if a new model string appears in `queue.jsonl` without being in the map.

## How much does the fragmentation cost in dollars?

Rough back-of-envelope using the totals from the queue:

- Rows: 1,689
- Total input tokens: 3,557,732,345 (~3.56B)
- Total cached_input tokens: 6,462,270,294 (~6.46B). The cached number exceeds the input number because cached is reported separately and *adds*, not divides — see the cache-ratio caveat below.
- Total output tokens: 47,254,694
- Total reasoning_output tokens: 1,098,634

If 29.6% of gpt-5-family rows were misattributed to other lines (the gap between 535 and 760), and gpt-5-family rows on this device average something close to the 633K input the sample row shows, the dollar misattribution is roughly:

```
225 misattributed hour-buckets × ~600K input tokens × ~$0.001/Ktok blended
≈ $135 of cost reported against the wrong model name
```

Tiny in absolute terms on this machine. But scale it to a team of 50 engineers each running CLIs at this rate, and you're looking at **~$6,750/month of cost attributed to the wrong line item on the budget**. The story is not the dollars; the story is that your "optimize the most expensive model" decision was made on a histogram bar that was 30% wrong.

## A note on the cache_ratio number

A naïve `cached / input` over the whole queue gives **1.8164** — a "ratio" greater than 1, which is nonsense if you read it as a percentage. The reason: the per-row schema reports `cached_input_tokens` as a *separate counter*, not a subset. Many CLIs count cached reads as additional billed input (because the API charges for them, just at a discount). When you sum across rows, the cached counter outpaces the un-cached counter because heavy reuse sessions push it past 1×. The correct interpretation is "for every uncached input token, the device also paid for ~1.8 cached input tokens." That is a separate post; mentioning it here only because *it is also a fragmentation problem* — different CLIs report this field differently, and any aggregation is suspect until you pin down the meaning per-source.

## The proposed normalizer

A one-file Python module, ~60 lines, that takes a model string and returns `(family, version, sku, source_prefix)`:

```python
def normalize(model: str) -> tuple[str, str, str, str | None]:
    s = model.strip()
    prefix = None
    # Strip known route prefixes
    for p in ("github_copilot/", "github.copilot-chat/"):
        if s.startswith(p):
            prefix = p.rstrip("/")
            s = s[len(p):]
            break
    # Family extraction
    s_low = s.lower()
    if s_low.startswith("gpt-5") or s_low == "gpt-5":
        return ("gpt5", _ver_after("gpt-5", s_low), "default", prefix)
    if "claude-opus-4" in s_low:
        ver = "4.7" if "4.7" in s_low or "4-7" in s_low else \
              "4.6" if "4.6" in s_low or "4-6" in s_low else "unknown"
        sku = "1m" if s_low.endswith("-1m") else "default"
        return ("opus", ver, sku, prefix)
    if "claude-sonnet" in s_low:
        ver = "4.6" if "4.6" in s_low or "4-6" in s_low else \
              "4.5" if "4.5" in s_low else \
              "4"   if s_low.endswith("-4")   else "unknown"
        return ("sonnet", ver, "default", prefix)
    if "claude-haiku" in s_low:
        return ("haiku", "4.5", "default", prefix)
    if "gemini" in s_low:
        return ("gemini", "3-pro-preview", "default", prefix)
    if s_low == "big-pickle":
        return ("CODENAME", "big-pickle", "default", prefix)
    return ("unknown", s, "default", prefix)
```

CI rule: every value of `model` seen in `queue.jsonl` over the last 7 days must round-trip through `normalize` without landing in the `unknown` family. If it doesn't, the build fails and a human has to either (a) extend the regex, or (b) add the codename to the allowlist. This is the only way to stop the histogram from drifting silently.

## The lesson

Telemetry pipelines treat `model` as a primary key. They shouldn't. **`model` is a free-text label written by whichever CLI happened to log the row**, and CLIs disagree about almost everything: dot vs. dash, point version vs. base, prefixed-by-route vs. unprefixed, SKU suffix vs. no suffix, internal codename vs. public name. Twenty strings for five families on a single user's machine.

The fix is unglamorous: a normalizer, an allowlist, and a CI test. The cost of *not* doing it is that every "which model is most expensive / fastest / cachiest" decision you make is wrong by 10–50%, depending on the family. And you won't notice, because the chart still has bars and the bars are still in some kind of order, and the largest bar is still labeled in a way that *sounds* like a real model name. It just isn't the real one.
